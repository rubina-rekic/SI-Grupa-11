# Sprint Goal – Sprint 4

## Sprint broj
4

## Sprint cilj
Cilj Sprinta 4 je završna validacija human-first faze projekta i uspostavljanje
tehničkih temelja za nadolazeću implementacijsku fazu. Tim definiše Definition
of Done kao zajednički standard kvaliteta, izrađuje Initial Release Plan koji
povezuje backlog sa sprintovima do kraja semestra, te postavlja tehnički skeleton
sistema — repozitorij, osnovnu strukturu projekta i inicijalni tehnički setup
backenda i frontenda — kako bi naredni sprintovi mogli započeti implementaciju
bez infrastrukturnih prepreka.

## Ključne stavke koje tim želi završiti
- [ ] Definition of Done – jedinstveni kriteriji završenosti za sve tipove
      radnih stavki (kod, dokumentacija, test, review)
- [ ] Initial Release Plan – mapiranje prioritetnog backloga na preostale
      sprintove sa jasnim milestoneovima i release kandidatima
- [ ] Osnovni skeleton projekta – inicijalna struktura backend (ASP.NET Core
      API/BLL/DAL) i frontend (React + Vite + TypeScript) aplikacije u skladu
      sa Architecture Overviewom
- [ ] Inicijalna struktura repozitorija i osnovni tehnički setup – Git
      repozitorij, branch strategija, README, osnovne dev skripte i
      konfiguracija za lokalno pokretanje

## Rizici i zavisnosti
- Definition of Done mora biti usklađena sa Test Strategijom i
  Architecture Overviewom iz Sprinta 3 kako bi kriteriji bili provjerljivi
- Initial Release Plan zavisi od prioritizovanog Product Backloga i
  realno procijenjenih kapaciteta tima u preostalim sprintovima
- Tehnički skeleton ne smije donositi skrivene arhitekturne odluke koje
  nisu evidentirane u Architecture Overviewu / Decision Logu
- Kašnjenje u postavljanju skeletona direktno blokira implementacijske
  sprintove 5–9 (rizik R-011 u Risk Registeru)
- Svi članovi tima moraju moći samostalno klonirati i pokrenuti projekat
  lokalno prije završetka sprinta (rizik R-020 — razlike između okruženja)
